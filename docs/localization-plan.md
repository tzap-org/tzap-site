# Localization plan

## Policy

English is the canonical source locale. Simplified Chinese and Traditional
Chinese are maintained first-party locales. New languages should start as
machine-assisted drafts and only be published after a review pass.

## Current locales

- `en`: English, served from `/`
- `zh-Hans`: Simplified Chinese, served from `/zh-Hans/`
- `zh-Hant`: Traditional Chinese, served from `/zh-Hant/`
- `ja`: Japanese, machine-generated draft
- `ko`: Korean, machine-generated draft
- `de`: German, machine-generated draft
- `fr`: French, machine-generated draft
- `es`: Spanish, machine-generated draft
- `pt-BR`: Brazilian Portuguese, machine-generated draft

## Translation workflow

1. Update `src/i18n/en.ts` first.
2. Generate draft files with `npm run translate:draft -- <locale>`.
3. Review or replace the generated string values.
4. Do not translate commands, URLs, package names, product names, file
   extensions, or identifiers such as `tzap`, `ZManager CLI`, `Zstandard`,
   `Reed-Solomon`, `Cargo`, `Homebrew`, `Open Collective`, and `Ko-fi`.
5. Review cryptography, recovery, archive-format, and installation wording by a
   human before publishing.
6. Register the locale in `src/i18n/index.ts`.
7. Add navigation labels in `src/data/site.ts`.
8. Run `npm run build`.
9. Check the generated pages for layout overflow, especially language switchers,
   buttons, headings, and support cards.

## Implementation notes

The translation schema is derived from `src/i18n/en.ts`. Other locale modules
use `satisfies SiteCopy`, so TypeScript catches missing or misspelled keys.

Localized pages are generated by `src/pages/[...path].astro` using `routeSlugs`
and `locales`, so adding a language should not require duplicating page files.

`scripts/generate-draft-locales.mjs` uses the public Google Translate endpoint
for draft generation. It protects common product names and leaves command
strings untouched, but its output is still only a starting point.
